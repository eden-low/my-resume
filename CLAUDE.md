# CLAUDE.md

Guidance for Claude Code when working in this repo. See [README.md](README.md) for the project overview.

## What this is

A static, 12-page HTML personal system ("Personal OS") for Low Fang Jun, styled as a dark glassmorphism "system dashboard" (near-black canvas, translucent blurred cards, one soft violet accent, Apple system fonts). No build tools, no JS framework, no package.json — just plain HTML/CSS files opened directly in a browser or served statically. The whole site is **login-first**: every page requires Google sign-in via `login.html` before it renders (see Architecture below) — there is no public-facing page anymore, including what used to be the public resume/contact pages. Firebase (Auth/Firestore, and Storage for gallery/journal) is loaded via ES module imports straight from `gstatic.com`, and the homepage additionally calls the OpenWeatherMap REST API and the browser Geolocation API — still buildless: no npm, no bundler. The site is installable as a PWA (`manifest.json` + `service-worker.js`, real `images/icon-192.png`/`icon-512.png` app icons).

**Design history**: the site went through two earlier looks — first a neon-purple "Solo Leveling hunter status" theme (glowing borders, Orbitron display font, RPG vocabulary like "quest"/"guild"/"inventory"), then a copy-only cleanup that kept the neon visuals but reworded the RPG terms into professional ones (Quest Board → Education, etc. — `resume.html`'s section `id`s still carry the old short names). A later pass replaced the *visual* language too: neon glow → glass blur, hot magenta accent → soft violet, Orbitron/Fira Code webfonts → system UI/monospace fonts, grid-paper background → soft ambient radial glow. `resume.html`'s copy from the second pass is unchanged; only the shared tokens it's built from were re-valued. The site was then converted from a public portfolio into a login-first personal system: a site-wide auth gate (`auth-guard.js`), a dedicated `login.html`, a `settings.html` (profile/preferences/login history/access management), login-history logging (`login_logs` Firestore collection), a basic light-mode toggle, and PWA scaffolding were added. Most recently ("Personal OS v1.2"), five features were layered on top: real PWA icons, an owner-only Export & Backup section in Settings, a Habit Tracker (`habits.html`), gallery social features (likes/comments/owner-only view analytics), and an owner-only Notification Center (`notifications.html`) — see the new/updated bullets below.

## Architecture

- **No shared layout/include system.** Every page (`index.html`, `resume.html`, `gallery.html`, `journal.html`, `expenses.html`, `timeline.html`, `habits.html`, `dashboard.html`, `notifications.html`, `contact.html`, `settings.html`) is a fully standalone HTML file that repeats the same `<head>` (Tailwind CDN, Font Awesome, `styles.css`, PWA manifest/theme-color/apple-touch-icon tags, a theme-preload inline `<script>` — see Light mode below) and the same header/nav markup (11 links plus an unread-notification badge next to "Notifications"). `login.html` is the one page that intentionally does **not** follow the shared nav/auth-guard pattern — see below.
- **Site-wide login gate.** [auth-guard.js](auth-guard.js) is a shared ES module — every protected page loads it via a single `<script type="module" src="auth-guard.js"></script>` tag (placed right after `scripts.js`, no other per-page wiring needed). It calls `onAuthStateChanged`; if there's no user it redirects to `login.html?redirect=<currentPage>`, otherwise it removes the `auth-check-pending` class from `<body>` to reveal the page (that class is added statically in each protected page's `<body class="...">` and maps to `visibility: hidden` in `styles.css`, so there's no flash of gated content before the redirect fires). It also forces a reload on a bfcache `pageshow` (`event.persisted`) so Back-navigation always re-checks auth instead of trusting a cached DOM. This is a UX convenience only — real access control is still enforced by `firestore.rules`/`storage.rules`, since page source is always viewable via view-source regardless. It also owns the one piece of genuinely cross-page UI: if the signed-in user `isOwner` and the page's nav has a `#notif-badge` element (every page's Notifications link does), it queries `notifications` for `uid == owner && read == false` and lights up the badge — a deliberate, minimal exception to "no shared files besides `scripts.js`," justified because the badge has to run identically on every single page.
- **[login.html](login.html)** is the one page every visitor can reach while signed out. It has its own inline `<script type="module">` (not `auth-guard.js`, since its logic is inverted): if already signed in it redirects away immediately (to `?redirect=` or `index.html`); otherwise it reveals a single "Sign in with Google" button. On a successful `signInWithPopup`, it writes a `login_logs` doc (`{ uid, email, loginTime: serverTimestamp(), device: navigator.userAgent, page: "login" }`), and — only if the signing-in user `isOwner` — a `notifications` doc ("New login detected from ..."), before redirecting. A `signingIn` flag guards against `onAuthStateChanged`'s own redirect firing mid-click and tearing down the page before these writes finish (that race was a real bug: the writes would get silently cancelled by navigation, with no console error). The `?redirect=` query param is validated against a strict `^[a-zA-Z0-9_-]+\.html$` regex to prevent an open-redirect vector.
- **Tailwind is loaded via CDN** (`cdn.tailwindcss.com`) and configured inline in a `<script>` block on every page — the same `tailwind.config` is copy-pasted into each file. Current token values (keep these in sync across all 12 pages if the palette changes): `darkBg:'#0a0a0e'`, `cardBg:'#17151f'`, `borderNeon:'#2a2833'`, `neonPurple:'#a78bfa'`, `neonBlue:'#6ea8fe'`, `neonViolet:'#8b7cf0'`, `textGray:'#9793ab'`; fonts are system stacks (`cyber`/`sans` → `-apple-system, BlinkMacSystemFont, "SF Pro Display/Text", "Segoe UI", sans-serif`; `code` → `ui-monospace, "SF Mono", Menlo, Consolas, "Fira Code", monospace`) — no webfont `<link>` tag needed or present.
- **[styles.css](styles.css)** holds the few things Tailwind utility classes can't express directly. Class *names* are unchanged from the old neon theme but now produce the glass look: `.neon-border-purple` is a translucent-border + `backdrop-filter: blur()` + soft drop-shadow card treatment (no colored glow), `.neon-text-purple`/`.neon-text-blue` are no-ops (`text-shadow: none`), `.grid-bg` renders a soft ambient radial-gradient glow instead of a grid-paper texture, plus custom scrollbar styling, the `.reveal`/`.reveal-group` scroll-animation classes, and the `[data-parallax-layer]` transition (see below).
- **[scripts.js](scripts.js)** is the one shared JS file (loaded via `<script src="scripts.js" defer></script>` near the end of `<body>` on every page). It uses `IntersectionObserver` to add `.is-visible` to any `.reveal` element as it scrolls into view (respecting `prefers-reduced-motion`), and also has a guarded mouse-parallax-tilt block for a `[data-parallax-hero]`/`[data-parallax-layer]` hero — currently unused since `index.html` no longer has a photo hero (it's a dashboard now), so this block is dead code that no-ops everywhere; safe to remove if the hero pattern never comes back. This file is the one exception to "no shared files" — keep new cross-page behavior here rather than inlining duplicate `<script>` blocks.
- **index.html** is a dashboard, not a hero page: a slim identity strip (small circular avatar using `images/profile.jpeg`, name, role), then a 12-col responsive grid with three glass cards — a System Status panel (name/role/focus/status, plus a live "Session" pill driven by an inline `<script type="module">` that imports `auth` from `firebase-init.js` and calls `onAuthStateChanged`), a live Weather widget (OpenWeatherMap Current Weather API; the API key is hardcoded client-side in the same inline module script, same trust model as the Firebase config), and a "Today's Habits" widget (checklist of today's check-in state per habit + a best-streak "🔥 N days" readout). The weather widget first tries the browser Geolocation API (`navigator.geolocation.getCurrentPosition`, an 8s timeout) and queries OpenWeatherMap by lat/lon if granted, falling back to a hardcoded `q=Kuching,MY` query if denied/unsupported/timed out — this only works over HTTPS or `localhost` (a browser security requirement for Geolocation), degrading silently to the Kuching fallback elsewhere. Below that grid sits an owner-only "Recent Notifications" card (last 5 unread, hidden entirely for non-owners), then quick-link cards to Resume/Gallery/Journal/Expenses/Timeline/Habits/Dashboard/Notifications/Contact/Settings in their own separate responsive grid (`grid-cols-2 sm:grid-cols-3 lg:grid-cols-5`, not tied to the 12-col system above) specifically so adding/removing a link is just adding/removing a card — no column-span math to recalculate. The habits/notifications widgets are self-contained inline `<script type="module">` blocks that duplicate a minimal version of `habits.js`/`notifications.js`'s fetch logic rather than importing from them, matching this codebase's per-page fetch/cache convention.
- **resume.html** combines what used to be 6 separate pages into one long page — Profile, Matrix, Education (`id="quests"`), Leadership & Events (`id="events"`), Work Experience (`id="experience"`), Achievements & Skills (`id="inventory"`) — each as a `<section id="...">` with `scroll-mt-24` so anchor jumps clear the sticky sub-nav. The section `id`s still use their original short names even though the visible headings were reworded away from RPG terms (Quest Board → Education, Guild Raid Archives → Leadership & Events, Gold Farming Quests → Work Experience, Achievements & Inventory → Achievements & Skills) — don't rename the `id`s without also updating the sub-nav anchor links. A sticky pill sub-nav (`sticky top-4`) sits under the header with anchor links to each section — this is in-page navigation only, not a routing system. It additionally loads Chart.js (`cdn.jsdelivr.net/npm/chart.js`) for the Matrix section's charts.
- **[images/](images/)** holds photo assets (`me1.jpeg`–`me5.jpeg`, `profile.jpeg` — the `index.html` identity-strip avatar) plus the real PWA icons `icon-192.png`/`icon-512.png` (dark `#09090e` rounded square with a violet "OS" monogram, generated via a one-off PowerShell/`System.Drawing` script — no build tool involved, just static files checked into the repo). `gallery.html` doesn't read from this folder — its photos are runtime data from Firebase Storage/Firestore (see below).
- **[firebase-init.js](firebase-init.js)** is the shared Firebase bootstrap (ES module): initializes the app from the project's config, calls `setPersistence(auth, browserLocalPersistence)` explicitly (defensive — guarantees the session survives a PWA relaunch rather than relying on the SDK default), and exports `auth`, `googleProvider`, `db`, `storage`, `OWNER_EMAIL`, and `isOwner(user)`. Every page/module that touches Firebase (`auth-guard.js`, `login.html`, `settings.js`, `export.js`, `gallery.js`, `expenses.js`, `journal.js`, `timeline.js`, `habits.js`, `dashboard.js`, `notifications.js`, and `index.html`'s inline auth-status module) imports from here rather than re-initializing Firebase.
- **[settings.html](settings.html) / [settings.js](settings.js)** is the Profile/Preferences/System Logs/Access Management/Export & Backup page. Profile is read-only off `auth.currentUser` (name/email/avatar/`metadata.creationTime`/`lastSignInTime`) plus the one Sign Out button in the whole site (`signOut(auth)` → redirect to `login.html`). Preferences (`theme`, `defaultCity`, `defaultVisibility`) are stored client-side in `localStorage` under the `lfj:settings` key — there's no per-user Firestore settings doc; the theme choice takes effect on the next reload (see Light mode below), and `defaultCity`/`defaultVisibility` are stored but not yet wired into the weather widget/compose-modal defaults on other pages (a small, optional follow-up per consuming page). System Logs, Access Management, and Export & Backup only render when `isOwner(auth.currentUser)`: System Logs is the last-20 `login_logs` docs (`orderBy("loginTime","desc"), limit(20))` — the one place in the codebase using Firestore `orderBy`, safe here because there's no paired `where` clause, so it only needs an automatic single-field index, not a composite one; Access Management does two full collection reads (`login_logs` reduced client-side to distinct emails by most-recent login, and `allowedUsers`) to render a Promote/Demote table that writes/deletes `allowedUsers/{email}` docs directly — this is the in-app replacement for what used to be a manual Firebase-Console-only workflow, and the owner's own row is never shown with Promote/Demote controls (they already bypass `allowedUsers` in every rule); Export & Backup is handled entirely by [export.js](export.js) (see below), `settings.js` just unhides the section.
- **[export.js](export.js)** (ES module, loaded only by `settings.html`) reuses `dashboard.js`'s `fetchCollection(name)` pattern (public-always + private-if-authorized, sorted client-side) for `expenses`, `journals`, `life_events`, `photos`, and `habits`, and a shared `downloadFile(filename, content, mimeType)` helper (Blob + `URL.createObjectURL` + a throwaway `<a download>` click — no library). Expenses export as CSV (`date,amount,category,note`); Journal exports as one combined Markdown file (`# Title` / date / mood / tags / blank line / content, per entry, joined with `---` separators); Timeline and Gallery metadata export as JSON (Firestore `Timestamp`s converted to ISO strings first — they aren't otherwise JSON-serializable in a useful form); "Full Backup" bundles `{ profile, settings, expenses, journals, timeline, gallery_metadata, habits }` into one `personal_os_backup.json`, where `profile` is read straight off `auth.currentUser` and `settings` is the parsed `lfj:settings` localStorage value. Downloads reflect whatever the signed-in user can currently read (public + private-if-authorized), not a strict "only docs this uid created" filter — there's no per-document ownership model to filter by in this single-owner app (see the `habits`/`notifications` write-model note below).
- **[gallery.js](gallery.js)** (ES module, loaded only by `gallery.html`) owns all gallery behavior as a single Instagram-style feed (not category grids): `onAuthStateChanged` drives the sign-in/out UI; `fetchVisiblePosts()` queries `photos` docs (public always, private only when authorized) with simple single-field `where` queries, capturing `{ id: doc.id, ...doc.data() }` (this used to be a bug — plain `doc.data()` with no id, which made it impossible to address a specific post's subcollections; fixed as part of adding social features), merges and sorts them **client-side** by `uploadedAt` (deliberately avoiding a Firestore `orderBy`, which would need a composite index), and caches the merged array so the filter tab bar (`#feed-filters`, `.filter-tab[data-filter]`) re-renders from memory instead of re-querying on every click. The Private tab and the owner-only "New Post" button (`#new-post-btn`) + modal (`#post-modal`) only appear once `isOwner(user)`/the private query succeeds. Dynamically-inserted post cards get `is-visible` added directly in JS (not just relying on `.reveal`/`IntersectionObserver`) since they're created after `scripts.js`'s one-time scroll-reveal query on page load — the modal itself intentionally has no `.reveal` class so it isn't affected by that mechanism at all.
  - **Likes/comments/views** (`photos/{id}/likes`, `/comments`, `/views` subcollections) are fetched alongside each post (`attachSocialCounts()`, two extra `getDocs` per post — acceptable at this app's personal scale) and rendered as a like button + count, a comment count that toggles an inline expand/lazy-load panel (`expandedComments` Set, same click-to-expand idea as `journal.js`'s `expandedIds`, deliberately not a separate modal — "lightweight, not a social network"), and — owner-only — a toggleable "Analytics" panel (`expandedAnalytics` Set) showing total views, unique visitors, and up to 5 recent visitor emails. Likes use the liker's own `uid` as the subcollection doc ID (`photos/{id}/likes/{uid}`), which is what makes "one like per user" structural rather than just a rules check. Views are recorded once per session (`viewedThisSession` Set, resets on reload) for any signed-in non-owner viewer, never for the owner viewing their own gallery.
- **[expenses.js](expenses.js)** (ES module, loaded only by `expenses.html`) mirrors `gallery.js`'s fetch/cache/filter pattern for the `expenses` Firestore collection (no Storage involved — it's numbers, not files), plus renders two Chart.js instances (`#daily-chart` bar, `#category-chart` doughnut) from the full cached set every time it refetches; both `Chart` instances are explicitly `.destroy()`ed before re-creating to avoid Chart.js's "canvas already in use" error on re-render. The category color/label map (`CATEGORY_META`) is the single source of truth for both the chart colors and the list-row badge styling — update it in one place if categories change. After every refetch, owner-gated `checkExpenseAlert()` sums this month's total and — if it's over the hardcoded RM1000 threshold and no alert has been written for this `YYYY-MM` yet (deduped via a `localStorage` checkpoint, not a Firestore query) — writes a `notifications` doc.
- **[journal.js](journal.js)** (ES module, loaded only by `journal.html`) is the third implementation of the same fetch/cache/filter pattern, for the `journals` Firestore collection. `MOOD_META` is the single source of truth for moods — it's used to generate both the compose form's `<select>` options and the mood filter chips at load time (don't hardcode a mood list in the HTML). Entry content is stored as raw markdown; `journalCard()` shows a plain-text `snippet()` by default and swaps in `marked.parse(entry.content)` when a card is clicked (tracked via an `expandedIds` Set, keyed by a composite of `uid`+`createdAt`+`title` since docs don't have a stable client-side id from a `getDocs` query without also grabbing `doc.id`). There's no HTML sanitizer on the rendered markdown — acceptable only because `journals` writes are owner-only in the rules, so the only person who can inject markup is the owner themselves. Image attachment is optional: if a file is present, it uploads to `journal/{public,private}/...` in Storage (mirroring the gallery's Storage layout) before the Firestore doc is written; otherwise `imageUrl` is `null`. A client-side search box filters the cached array by title/content/tags substring match, composed with the visibility and mood filters (all three narrow the same cached list — no extra Firestore reads). After every refetch, owner-gated `checkJournalReminder()` writes a `notifications` doc if the newest entry is 3+ days old (or there are none), deduped per calendar day via `localStorage`.
- **[timeline.js](timeline.js)** (ES module, loaded only by `timeline.html`) is the fourth implementation of the fetch/cache/filter pattern, for a `life_events` Firestore collection (`type: career|education|travel|personal`). Unlike the other three pages, `date` is a user-picked `<input type="date">` converted with `Timestamp.fromDate()` on save — an editable point-in-time value, not `serverTimestamp()` — because life events are commonly logged after the fact (e.g. backfilling a graduation date). `renderTimeline()` groups the filtered+searched cached array by `date`'s year into `<section>`s with a `border-l` vertical connector; `TYPE_META` is the single source of truth for the four event-type badges (icon/color/label). The search box matches title/description text **or** an exact year string (typing "2024" filters to that year) via `matchesSearch()`. **Deliberately out of scope**: automatic aggregation of Gallery uploads / Expense milestones / Journal entries into the timeline, and matching Travel-type events to related photos/expenses — both were in early planning notes but are materially bigger features (cross-collection queries, a unified item shape) better scoped once the manual timeline has real data to justify them.
- **[habits.js](habits.js)** (ES module, loaded only by `habits.html`) is the fifth implementation of the fetch/cache/filter pattern, for a `habits` Firestore collection (`{ uid, title, icon, completedDates: [], visibility, createdAt }`). `completedDates` is an array of local (not UTC — `toDateKey()` builds `YYYY-MM-DD` from `getFullYear()`/`getMonth()`/`getDate()` deliberately, to avoid `toISOString()`'s UTC-based date shifting near midnight) date strings; a check-in toggles today's key via `arrayUnion`/`arrayRemove`. `computeStreak()` counts consecutive days backward from today, but anchors on yesterday if today isn't checked in yet so the streak doesn't visually reset to 0 the moment midnight passes and before the user has checked in — this is a "current streak," not "days since creation." The weekly strip is the trailing 7 days (not the calendar Mon–Sun week); the monthly ring is `conic-gradient`-based (CSS only, no Chart.js) showing completed-days-this-month over days-elapsed-this-month. Check-in is owner-gated like every other write in this app. After a successful check-in, owner-gated `checkStreakNotification()` writes a `notifications` doc when the new streak is a positive multiple of 30, deduped via a `localStorage` checkpoint per habit (so re-checking the same milestone in, say, a data-fix scenario doesn't re-notify).
- **[dashboard.js](dashboard.js)** (ES module, loaded only by `dashboard.html`) is **read-only** — no forms, no writes, no modal. `fetchCollection(name)` is a generic version of the other pages' fetch pattern (public always, private only when authorized) reused for `photos`, `expenses`, and `journals` in turn; each is aggregated client-side into the Gallery/Expense/Journal Analytics sections (counts, top category, `mostCommon()` helper for mode calculations, three Chart.js instances for monthly/category/weekly expense breakdowns — same `.destroy()`-before-recreate pattern as `expenses.js`). The System Status section reads `auth.currentUser.metadata.creationTime`/`lastSignInTime` directly (no Firestore involved) and does its own simple, non-geolocated `q=Kuching,MY` OpenWeatherMap call (unlike `index.html`'s weather widget, this one doesn't try Geolocation first — it's a secondary status readout, not the primary weather feature). Because this page never writes anywhere, it needed **no new Firestore or Storage rules**. (`export.js`'s `fetchCollection()` is a separate copy of this same pattern, extended to more collections — not imported from here, per the codebase's per-page duplication convention.)
- **[notifications.html](notifications.html) / [notifications.js](notifications.js)** is an **owner-only** page — non-owners see an access note, never the list, because the `notifications` collection's rules only allow the owner to read it at all. There's no create/edit UI here: every `notifications` doc is written elsewhere (see the generation points below) by the owner's own client. `notifications.js` fetches with a single `where("uid","==",user.uid)` (no `orderBy`, to avoid needing a composite index — same reasoning as `gallery.js`/`timeline.js`) and sorts client-side; `TYPE_META` maps `login`/`expense_alert`/`journal_reminder`/`habit_streak`/`gallery` to an icon/color. **Why notifications are owner-write-only, not cross-user-write**: this app has no backend/Cloud Functions, so nothing can push a notification the instant something happens elsewhere — a "someone liked your photo" alert has to be *discovered* by the owner's own client the next time it loads relevant data, not pushed by the liker. Making `notifications` cross-user-writable (so a liker could write a notification doc targeting the owner) would have been the only way to get closer to "real-time," but it would open the one genuinely new write-permission surface in an app that otherwise has exactly one writer for everything — not worth it for a best-effort, non-realtime feature. Concretely: `login.html` writes a `login` notification right after its `login_logs` write; `expenses.js`/`journal.js`/`habits.js` write their own notifications inline in their existing fetch/check-in flows (see those bullets); `gallery.js`'s owner branch compares each post's fetched like count against a `localStorage`-cached count per `postId` and notifies on increase. All five generation points dedupe via `localStorage` (not a Firestore existence check) to avoid re-notifying on every page load.
- **[firestore.rules](firestore.rules) / [storage.rules](storage.rules)** are the security-rules source of truth for the `photos` (plus its `likes`/`comments`/`views` subcollections), `expenses`, `journals`, `life_events`, `habits`, `allowedUsers`, `login_logs`, and `notifications` Firestore collections, and the `gallery/{public,private}/...` and `journal/{public,private}/...` Storage paths (expenses, life_events, habits, login_logs, notifications, and dashboard have no Storage rules — none of them upload files). `login_logs` allows any authenticated user to `create` their own doc (`request.auth.uid == request.resource.data.uid`) but only the owner can `read` (no `update`/`delete` — it's an immutable log); `notifications` is owner-read/write-only, full stop (see the notifications bullet above for why). `allowedUsers`'s `read` rule changed from `false` to owner-only so the Settings Access Management table can enumerate the collection (safe: other rules' `exists()`/`firestore.exists()` checks against `allowedUsers` are rules-engine-internal and unaffected by tightening that collection's own `read` rule). **Firestore subcollection rules do not inherit from the parent `match` block** — `photos/{photoId}/likes`, `/comments`, and `/views` each need their own explicit `match` block, which is why `firestore.rules` now defines a top-level `canReadPost(photoId)` helper function (the first rules function in this file) that mirrors `photos`' own read condition via `get()`, reused by all three subcollections so the visibility check isn't tripled inline. `likes/{uid}` uses the liker's own uid as the doc ID (so `allow write` just checks `request.auth.uid == uid`); `comments`/`views` are create-only (no `update`/`delete`) like `login_logs`. The owner's email is hardcoded in several places that must stay in sync: `firebase-init.js` (`OWNER_EMAIL`), `firestore.rules` (now eight top-level collections), and `storage.rules` (two path groups).
- **Deploying rules**: [firebase.json](firebase.json) + [.firebaserc](.firebaserc) (project `lfj-profolio`) let rules changes be pushed via the Firebase CLI instead of pasting into the Console. It's a dev-only tool, not installed globally and not part of the site's runtime — run it via `npx` (one-time `npx firebase-tools login` to authorize, then `npx firebase-tools deploy --only firestore:rules,storage` after any `firestore.rules`/`storage.rules` edit). This doesn't change the site itself being buildless — no npm install/bundler is needed to serve or develop the actual pages.
- **PWA**: [manifest.json](manifest.json) (`name: "Personal OS"`, `short_name: "PersonalOS"`, `display: standalone`, `#09090e` theme/background colors, real `images/icon-192.png`/`icon-512.png` PNG icons — no longer a photo reused as a fake icon) and [service-worker.js](service-worker.js) (a minimal network-first-with-cache-fallback strategy, registered from `scripts.js` on `window.load`; `CACHE` is version-stamped (`lfj-shell-v2`) — bump it whenever `PRECACHE`'s file list changes, to force old caches to be dropped on the next `activate`). The service worker explicitly bypasses `gstatic.com`, `googleapis.com`, `firebaseapp.com`, `openweathermap.org`, and the Tailwind/Font Awesome/Chart.js/marked CDN hosts — it must never intercept Auth/Firestore/Storage/weather/CDN requests, since those need to always hit the network live. `firebase-init.js` additionally calls `setPersistence(auth, browserLocalPersistence)` so the signed-in session reliably survives a PWA relaunch.
- **Light mode**: a simple, functional toggle (not a full redesign), set via Settings' Preferences and stored in `localStorage` (`lfj:settings.theme`). Every page's `<head>` starts with a synchronous inline script that reads that value and sets `data-theme="light"` on `<html>` before first paint (avoiding a flash of the wrong theme); `styles.css` then overrides colors under `html[data-theme="light"] ...` using attribute-substring selectors like `[class*="bg-cardBg"]` so Tailwind's opacity-suffixed utility classes (`bg-cardBg/90`, `bg-darkBg/60`, etc.) are all caught by one rule each, rather than needing a rule per opacity variant. Coverage is limited to the color classes actually in use today — a new component using an uncovered class will silently stay dark-styled until `styles.css`'s light-mode block is updated.

## Conventions to follow when editing

- **Nav links**: every page's `<nav>` lists all 11 non-login pages (Home, Resume, Gallery, Journal, Expenses, Timeline, Habits, Dashboard, Notifications, Contact, Settings) — `login.html` is intentionally excluded from the nav since it's the signed-out entry point, not a destination. The Notifications link additionally carries a `<span id="notif-badge">` (see `auth-guard.js` above). If you add a new page, add its link to the nav in **all** existing HTML files, not just the new one — and if it's a Firebase-backed feature people should be able to jump to from the homepage, add a quick-link card to `index.html`'s quick-links grid too. When editing this block identically across many files at once, a scripted find/replace (e.g. a small PowerShell loop matching the exact `<a href="...">` line text) is far less error-prone than 8+ individual manual edits — just watch for CRLF vs LF line endings if you're inserting a new line adjacent to a matched one.
- **New protected pages need three things**: the theme-preload inline `<script>` as the first line of `<head>`, the PWA `<link rel="manifest">`/`theme-color`/`apple-touch-icon` tags near the `styles.css` link, `auth-check-pending` added to the `<body class="...">` string, and `<script type="module" src="auth-guard.js"></script>` placed right after the `scripts.js` tag. Copying an existing protected page (e.g. `contact.html` or `timeline.html`) as the template carries all of this over automatically — see below.
- **Color palette**: `darkBg`, `cardBg`, `borderNeon`, `neonPurple`, `neonBlue`, `neonViolet`, `textGray` — defined identically in each page's inline `tailwind.config` (values listed in Architecture above). Keep them in sync if the palette changes; there's no single source of truth to edit.
- **Fonts**: `font-cyber` (system UI stack, headings/labels), `font-code` (system monospace stack, small tags/data/labels), `font-sans` (system UI stack, body text, default) — no external font loading.
- **Card style**: content blocks use `bg-cardBg/90 backdrop-blur-sm p-6 rounded-2xl neon-border-purple` (glass, not glow — see styles.css note above), often with `hover:-translate-y-1 transition-all` on clickable cards.
- **Icons**: Font Awesome 6 solid icons (`fa-solid fa-*`), colored per section (purple/blue/emerald/amber/rose) to visually distinguish categories.
- **Scroll reveal**: add class `reveal` to any top-level section or card that should fade/slide in on scroll. When a group of sibling cards should stagger in one after another, add `reveal-group` to their shared parent (the stagger delays are defined in styles.css as `.reveal-group > .reveal:nth-child(n)`). Every page must include `<script src="scripts.js" defer></script>` before `</body>` for this to work.
- When adding a new section/page, copy the closest existing page (same head + header/nav block) as the starting template rather than writing one from scratch, to keep everything consistent.
- Avoid meta-commentary text referencing "the original resume" or "preserved exactly as intended" in page copy — content should read as the resume itself, not as a description of the HTML conversion.
- **Copy tone**: the visual style (neon glow, cyber fonts, grid background) still nods to Solo Leveling, but page copy should read as a professional "system profile" — avoid game/RPG vocabulary (hunter, quest, guild, level/LV, EXP, inventory, loot-rarity tags like SSR/SR/R, weapon/scroll/grail flavor text, dungeon, etc.). Prefer plain dashboard/resume terms (Profile, Education, Leadership, Work Experience, Achievements, Role, Score).
