# CLAUDE.md

Guidance for Claude Code when working in this repo. See [README.md](README.md) for the project overview.

## What this is

A static, 5-page HTML portfolio/resume for Low Fang Jun, styled as a dark glassmorphism "system dashboard" (near-black canvas, translucent blurred cards, one soft violet accent, Apple system fonts). No build tools, no JS framework, no package.json — just plain HTML/CSS files opened directly in a browser or served statically. Three pages talk to Firebase (Auth/Firestore, and Storage for the gallery) via ES module imports straight from `gstatic.com`, and the homepage additionally calls the OpenWeatherMap REST API — still buildless: no npm, no bundler.

**Design history**: the site went through two earlier looks — first a neon-purple "Solo Leveling hunter status" theme (glowing borders, Orbitron display font, RPG vocabulary like "quest"/"guild"/"inventory"), then a copy-only cleanup that kept the neon visuals but reworded the RPG terms into professional ones (Quest Board → Education, etc. — `resume.html`'s section `id`s still carry the old short names). The current pass replaced the *visual* language too: neon glow → glass blur, hot magenta accent → soft violet, Orbitron/Fira Code webfonts → system UI/monospace fonts, grid-paper background → soft ambient radial glow. `resume.html`'s copy from the second pass is unchanged; only the shared tokens it's built from were re-valued.

## Architecture

- **No shared layout/include system.** Every page (`index.html`, `resume.html`, `gallery.html`, `expenses.html`, `contact.html`) is a fully standalone HTML file that repeats the same `<head>` (Tailwind CDN, Font Awesome, `styles.css`) and the same header/nav markup.
- **Tailwind is loaded via CDN** (`cdn.tailwindcss.com`) and configured inline in a `<script>` block on every page — the same `tailwind.config` is copy-pasted into each file. Current token values (keep these in sync across all 5 pages if the palette changes): `darkBg:'#0a0a0e'`, `cardBg:'#17151f'`, `borderNeon:'#2a2833'`, `neonPurple:'#a78bfa'`, `neonBlue:'#6ea8fe'`, `neonViolet:'#8b7cf0'`, `textGray:'#9793ab'`; fonts are system stacks (`cyber`/`sans` → `-apple-system, BlinkMacSystemFont, "SF Pro Display/Text", "Segoe UI", sans-serif`; `code` → `ui-monospace, "SF Mono", Menlo, Consolas, "Fira Code", monospace`) — no webfont `<link>` tag needed or present.
- **[styles.css](styles.css)** holds the few things Tailwind utility classes can't express directly. Class *names* are unchanged from the old neon theme but now produce the glass look: `.neon-border-purple` is a translucent-border + `backdrop-filter: blur()` + soft drop-shadow card treatment (no colored glow), `.neon-text-purple`/`.neon-text-blue` are no-ops (`text-shadow: none`), `.grid-bg` renders a soft ambient radial-gradient glow instead of a grid-paper texture, plus custom scrollbar styling, the `.reveal`/`.reveal-group` scroll-animation classes, and the `[data-parallax-layer]` transition (see below).
- **[scripts.js](scripts.js)** is the one shared JS file (loaded via `<script src="scripts.js" defer></script>` near the end of `<body>` on every page). It uses `IntersectionObserver` to add `.is-visible` to any `.reveal` element as it scrolls into view (respecting `prefers-reduced-motion`), and also has a guarded mouse-parallax-tilt block for a `[data-parallax-hero]`/`[data-parallax-layer]` hero — currently unused since `index.html` no longer has a photo hero (it's a dashboard now), so this block is dead code that no-ops everywhere; safe to remove if the hero pattern never comes back. This file is the one exception to "no shared files" — keep new cross-page behavior here rather than inlining duplicate `<script>` blocks.
- **index.html** is a dashboard, not a hero page: a slim identity strip (small circular avatar using `images/me3.jpeg`, name, role), then a 12-col responsive grid of glass cards — a System Status panel (name/role/focus/status, plus a live "Session" pill driven by an inline `<script type="module">` that imports `auth` from `firebase-init.js` and calls `onAuthStateChanged`), a live Weather widget (Kuching, OpenWeatherMap Current Weather API, the API key is hardcoded client-side in the same inline module script, refreshes every 15 minutes), and quick-link cards to Resume/Gallery/Expenses/Contact.
- **resume.html** combines what used to be 6 separate pages into one long page — Profile, Matrix, Education (`id="quests"`), Leadership & Events (`id="events"`), Work Experience (`id="experience"`), Achievements & Skills (`id="inventory"`) — each as a `<section id="...">` with `scroll-mt-24` so anchor jumps clear the sticky sub-nav. The section `id`s still use their original short names even though the visible headings were reworded away from RPG terms (Quest Board → Education, Guild Raid Archives → Leadership & Events, Gold Farming Quests → Work Experience, Achievements & Inventory → Achievements & Skills) — don't rename the `id`s without also updating the sub-nav anchor links. A sticky pill sub-nav (`sticky top-4`) sits under the header with anchor links to each section — this is in-page navigation only, not a routing system. It additionally loads Chart.js (`cdn.jsdelivr.net/npm/chart.js`) for the Matrix section's charts.
- **[images/](images/)** holds photo assets (`me1.jpeg`–`me4.jpeg`); only `me3.jpeg` is currently referenced (the `index.html` identity-strip avatar). `gallery.html` doesn't read from this folder — its photos are runtime data from Firebase Storage/Firestore (see below).
- **[firebase-init.js](firebase-init.js)** is the shared Firebase bootstrap (ES module): initializes the app from the project's config, exports `auth`, `googleProvider`, `db`, `storage`, `OWNER_EMAIL`, and `isOwner(user)`. `gallery.js`, `expenses.js`, and `index.html`'s inline auth-status module all import from here rather than re-initializing Firebase.
- **[gallery.js](gallery.js)** (ES module, loaded only by `gallery.html`) owns all gallery behavior as a single Instagram-style feed (not category grids): `onAuthStateChanged` drives the sign-in/out UI; `fetchVisiblePosts()` queries `photos` docs (public always, private only when authorized) with simple single-field `where` queries, merges and sorts them **client-side** by `uploadedAt` (deliberately avoiding a Firestore `orderBy`, which would need a composite index), and caches the merged array so the filter tab bar (`#feed-filters`, `.filter-tab[data-filter]`) re-renders from memory instead of re-querying on every click. The Private tab and the owner-only "New Post" button (`#new-post-btn`) + modal (`#post-modal`) only appear once `isOwner(user)`/the private query succeeds. Dynamically-inserted post cards get `is-visible` added directly in JS (not just relying on `.reveal`/`IntersectionObserver`) since they're created after `scripts.js`'s one-time scroll-reveal query on page load — the modal itself intentionally has no `.reveal` class so it isn't affected by that mechanism at all.
- **[expenses.js](expenses.js)** (ES module, loaded only by `expenses.html`) mirrors `gallery.js`'s fetch/cache/filter pattern for the `expenses` Firestore collection (no Storage involved — it's numbers, not files), plus renders two Chart.js instances (`#daily-chart` bar, `#category-chart` doughnut) from the full cached set every time it refetches; both `Chart` instances are explicitly `.destroy()`ed before re-creating to avoid Chart.js's "canvas already in use" error on re-render. The category color/label map (`CATEGORY_META`) is the single source of truth for both the chart colors and the list-row badge styling — update it in one place if categories change.
- **[firestore.rules](firestore.rules) / [storage.rules](storage.rules)** are the security-rules source of truth for the `photos`, `expenses`, and `allowedUsers` Firestore collections and the `gallery/{public,private}/...` Storage paths (expenses has no Storage rules — it never uploads files). There's no Firebase CLI/deploy wiring in this repo — after editing either file, paste its contents into the corresponding tab in the Firebase Console manually. The owner's email is hardcoded in four places that must stay in sync: `firebase-init.js` (`OWNER_EMAIL`), `firestore.rules` (two collections), and `storage.rules`.

## Conventions to follow when editing

- **Nav links**: every page's `<nav>` lists all 5 pages (Home, Resume, Gallery, Expenses, Contact). If you add a new page, add its link to the nav in **all** existing HTML files, not just the new one.
- **Color palette**: `darkBg`, `cardBg`, `borderNeon`, `neonPurple`, `neonBlue`, `neonViolet`, `textGray` — defined identically in each page's inline `tailwind.config` (values listed in Architecture above). Keep them in sync if the palette changes; there's no single source of truth to edit.
- **Fonts**: `font-cyber` (system UI stack, headings/labels), `font-code` (system monospace stack, small tags/data/labels), `font-sans` (system UI stack, body text, default) — no external font loading.
- **Card style**: content blocks use `bg-cardBg/90 backdrop-blur-sm p-6 rounded-2xl neon-border-purple` (glass, not glow — see styles.css note above), often with `hover:-translate-y-1 transition-all` on clickable cards.
- **Icons**: Font Awesome 6 solid icons (`fa-solid fa-*`), colored per section (purple/blue/emerald/amber/rose) to visually distinguish categories.
- **Scroll reveal**: add class `reveal` to any top-level section or card that should fade/slide in on scroll. When a group of sibling cards should stagger in one after another, add `reveal-group` to their shared parent (the stagger delays are defined in styles.css as `.reveal-group > .reveal:nth-child(n)`). Every page must include `<script src="scripts.js" defer></script>` before `</body>` for this to work.
- When adding a new section/page, copy the closest existing page (same head + header/nav block) as the starting template rather than writing one from scratch, to keep everything consistent.
- Avoid meta-commentary text referencing "the original resume" or "preserved exactly as intended" in page copy — content should read as the resume itself, not as a description of the HTML conversion.
- **Copy tone**: the visual style (neon glow, cyber fonts, grid background) still nods to Solo Leveling, but page copy should read as a professional "system profile" — avoid game/RPG vocabulary (hunter, quest, guild, level/LV, EXP, inventory, loot-rarity tags like SSR/SR/R, weapon/scroll/grail flavor text, dungeon, etc.). Prefer plain dashboard/resume terms (Profile, Education, Leadership, Work Experience, Achievements, Role, Score).

When adding a new page, section, or notable structural change (e.g. introducing a shared layout, a build step, or a new palette), update both [README.md](README.md) (page table / tech stack) and this file (architecture / conventions) so they stay accurate.
